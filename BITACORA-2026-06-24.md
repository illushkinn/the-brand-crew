## 📱 Filosofía de Responsive Design

### Principios Establecidos

#### 1. Mobile-First Approach
- Diseñar primero para pantallas pequeñas (320px-375px)
- Expandir progresivamente a tablets y desktop
- No asumir que "mobile = iPhone 12+"

#### 2. Fluid Layouts Over Fixed
- **Usar:** `%`, `vw`, `vh`, `clamp()`, `min()`, `max()`
- **Evitar:** Valores fijos en px (excepto para borders, small gaps)
- **Ejemplo:** `font-size: clamp(1rem, 2.5vw, 1.5rem)`

#### 3. CSS Moderno
- **`clamp(min, preferred, max)`** - Valores fluidos con límites
- **CSS Grid + Flexbox** - Layouts robustos
- **Container Queries** - Para componentes verdaderamente responsive
- **Logical Properties** - `margin-inline`, `padding-block`

#### 4. Media Queries Estratégicos
- **Basados en contenido, no dispositivos**
  - ❌ `@media (max-width: 375px)` (iPhone SE específico)
  - ✅ `@media (max-width: 400px)` (cuando el contenido se rompe)
- **Combinar width + height para casos edge:**
  - `@media (max-height: 700px) and (max-width: 400px)` - iPhone SE portrait
  - `@media (orientation: landscape) and (max-height: 500px)` - Landscape pequeño

#### 5. Testing Constante
- **Chrome DevTools Device Emulation:**
  - iPhone SE (375x667px) - El más pequeño común
  - iPhone 12 Mini (375x812px)
  - iPhone 14 Pro Max (430x932px)
  - Samsung Galaxy S8+ (360x740px)
- **Dispositivos reales cuando sea posible**
- **Responsive Design Mode:** Probar anchos arbitrarios (320px-1920px)

#### 6. Viewport Units Awareness
- **`vh` problems:** No incluye browser chrome en mobile
- **Solución:** Usar `dvh` (dynamic vh) o `svh` (small vh)
  ```css
  height: 100vh; /* fallback */
  height: 100dvh; /* preferred */
  ```
- **Para hero sections:** Preferir `min-height` sobre `height` para permitir scroll si contenido excede

### Problemas Identificados (Pendientes)

#### Issue #1: iPhone SE Hero Black Space
**Problema:** 1/4 de pantalla en negro después del pricing en iPhone SE
**Causa probable:**
- Hero usa `height: 100vh` fijo
- Contenido (título + subtitle + meta + pricing + actions) excede 667px
- No hay espacio para scroll

**Solución propuesta:**
```css
.hero {
  min-height: 100vh; /* Permite crecer si es necesario */
  min-height: 100dvh;
}

/* Compactar en iPhone SE portrait */
@media (max-height: 700px) and (max-width: 400px) {
  .hero-title { font-size: clamp(2rem, 6vw, 2.5rem); }
  .hero-sub { font-size: clamp(0.95rem, 2.5vw, 1.2rem); }
  .hero-meta { font-size: 0.75rem; gap: 0.25rem; }
  .hero-actions .btn-lg { padding: 0.65rem 1.2rem; font-size: 0.95rem; }
}
```

## 🎓 Lecciones Aprendidas

1. **DNS/Vercel Domain Issues:**
   - Los commits pueden no ser la causa de domain issues
   - Siempre verificar Vercel Dashboard domain configuration
   - Error 308 = redirect permanente, usualmente config issue
   - Namecheap BasicDNS más simple que cPanel DNS

2. **Responsive Design:**
   - iPhone SE (375x667px) es el bottleneck más común
   - Combinar media queries de width + height captura edge cases
   - `min-height` mejor que `height` fijo para hero sections
   - `clamp()` es tu amigo para responsive typography

3. **Git Workflow:**
   - Force push puede ser necesario para crisis de production
   - Commits vacíos pueden trigger rebuilds pero no siempre funcionan
   - Cache bust real (cambio en archivo) más efectivo
